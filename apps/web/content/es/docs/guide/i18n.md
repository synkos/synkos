Synkos usa [`vue-i18n`](https://vue-i18n.intlify.dev/) y trae strings core en inglés (`en-US`) y español (`es-ES`) — los labels que ves en settings, gestión de cuenta, el flujo de auth y los componentes built-in. Tú añades tus traducciones encima.

## Cómo funciona el merge

`createSynkosBoot({ messages })` mergea tus mensajes con los mensajes core del framework. Tus claves ganan en conflicto, así que puedes sobreescribir cualquier string del framework redefiniendo su clave en tu propio bundle.

```ts
// src/boot/synkos.ts
import { createSynkosBoot } from '@synkos/client';
import config from '../app.config';

import en from '../i18n/en-US';
import es from '../i18n/es-ES';

export default createSynkosBoot({
  config,
  messages: { 'en-US': en, 'es-ES': es },
});
```

## Estructura de archivos

Tus traducciones viven junto a las del framework:

```
src/i18n/
├── en-US/
│   ├── index.ts        ← tus strings (re-export)
│   ├── home.ts
│   └── settings.ts     ← sobreescribe cualquier clave del framework aquí
└── es-ES/
    └── ...
```

Cada módulo exporta un objeto plano. El CLI scaffoldea un árbol de ejemplo.

## Usar strings en templates

```vue
<script setup lang="ts">
import { useI18n } from 'vue-i18n';
const { t } = useI18n();
</script>

<template>
  <h1>{{ t('home.title') }}</h1>
  <AppButton>{{ t('common.continue') }}</AppButton>
</template>
```

En `<template>`, prefiere `{{ t('key') }}` sobre el shortcut `$t` — es idéntico, está tipado y se le pueden meter reglas de ESLint encima más tarde.

## Convención de naming

Las claves del framework viven bajo namespaces top-level (`auth.*`, `settings.*`, `account.*`, `notifications.*`). Usa tus propios namespaces para strings específicos de la app (`home.*`, `inbox.*`, `dashboard.*`).

```ts
// src/i18n/en-US/home.ts
export default {
  home: {
    title: 'Welcome back',
    greeting: 'Hello, {name}',
    cta: 'Get started',
  },
};
```

Las claves van en camelCase. Evita usar frases como claves — usa un identificador corto.

## Pluralización

Usa la sintaxis estándar de pipe de `vue-i18n`:

```ts
{
  inbox: {
    unreadCount: 'no unread | 1 unread | {count} unread',
  },
}
```

```vue
<span>{{ t('inbox.unreadCount', count) }}</span>
```

## Interpolación

```ts
{
  home: {
    greeting: 'Hello, {name}';
  }
}
```

```vue
<p>{{ t('home.greeting', { name: user.firstName }) }}</p>
```

## Cambiar de idioma

Usa el settings store — persiste la elección y re-renderiza todo:

```ts
import { useSettingsStore } from '@synkos/client';

const settings = useSettingsStore();
settings.appLang = 'es-ES';
```

El cambio es instantáneo. La locale de la app también se propaga a los plugins nativos (formateo de fechas, idioma de push notifications).

## Añadir un idioma nuevo

1. Crea `src/i18n/<locale>/index.ts` con tus traducciones
2. Pásalo a `createSynkosBoot({ messages })`
3. Añade la locale a `app.config.ts`:

```ts
defineAppConfig({
  i18n: {
    locales: [
      { code: 'en-US', name: 'English' },
      { code: 'es-ES', name: 'Español' },
      { code: 'fr-FR', name: 'Français' }, // ← nuevo
    ],
  },
});
```

El selector de idioma en settings lo recogerá automáticamente.

## Qué pasa cuando falta una clave

En dev: warning de consola + la clave se renderiza tal cual.
En prod: fallback silencioso a la locale por defecto, clave renderizada si también falta ahí.

Recomendamos correr con el handler de claves perdidas de `vue-i18n` en dev para detectar problemas pronto.

## Anti-patrones

**No hardcodees strings visibles al usuario.** Incluso los strings de un solo uso deberían pasar por `t()`. El coste es una línea por string — el beneficio es poder lanzar a un mercado nuevo sin un cambio de código.

**No metas HTML en traducciones.** Usa interpolación de slots:

```vue
<i18n-t keypath="legal.agreeTo" tag="p">
  <template #terms>
    <a href="/terms">{{ t('legal.terms') }}</a>
  </template>
</i18n-t>
```

```ts
{ legal: { agreeTo: 'Acepto los {terms}', terms: 'términos' } }
```
