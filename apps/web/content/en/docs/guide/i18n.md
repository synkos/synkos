Synkos uses [`vue-i18n`](https://vue-i18n.intlify.dev/) and ships with English (`en-US`) and Spanish (`es-ES`) core strings — the labels you see in settings, account management, the auth flow, and built-in components. You add your own translations on top.

## How merging works

`createSynkosBoot({ messages })` merges your messages with the framework's core messages. Your keys win on conflict, so you can override any framework string by redefining its key in your own bundle.

```ts
// src/boot/synkos.ts
import { createSynkosBoot } from '@synkos/client';
import config from '../app.config';

import en from '../i18n/en-US';
import es from '../i18n/es-ES';

export default createSynkosBoot({
  config,
  messages: { 'en-US': en, 'es-ES': es },
});
```

## File layout

Your translations live alongside the framework's:

```
src/i18n/
├── en-US/
│   ├── index.ts        ← your strings (re-export)
│   ├── home.ts
│   └── settings.ts     ← override any framework key here
└── es-ES/
    └── ...
```

Each module exports a plain object. The CLI scaffolds an example tree.

## Using strings in templates

```vue
<script setup lang="ts">
import { useI18n } from 'vue-i18n';
const { t } = useI18n();
</script>

<template>
  <h1>{{ t('home.title') }}</h1>
  <AppButton>{{ t('common.continue') }}</AppButton>
</template>
```

In `<template>`, prefer `{{ t('key') }}` over the `$t` shortcut — it's identical, it's typed, and ESLint rules can be tightened around it later.

## Naming convention

Framework keys live under top-level namespaces (`auth.*`, `settings.*`, `account.*`, `notifications.*`). Use your own namespaces for app-specific strings (`home.*`, `inbox.*`, `dashboard.*`).

```ts
// src/i18n/en-US/home.ts
export default {
  home: {
    title: 'Welcome back',
    greeting: 'Hello, {name}',
    cta: 'Get started',
  },
};
```

Keys are camelCase. Avoid sentences as keys — use a short identifier.

## Pluralization

Use the standard `vue-i18n` pipe syntax:

```ts
{
  inbox: {
    unreadCount: 'no unread | 1 unread | {count} unread',
  },
}
```

```vue
<span>{{ t('inbox.unreadCount', count) }}</span>
```

## Interpolation

```ts
{
  home: {
    greeting: 'Hello, {name}';
  }
}
```

```vue
<p>{{ t('home.greeting', { name: user.firstName }) }}</p>
```

## Switching languages

Use the settings store — it persists the choice and re-renders everything:

```ts
import { useSettingsStore } from '@synkos/client';

const settings = useSettingsStore();
settings.appLang = 'es-ES';
```

The change is instant. The app's locale also propagates to native plugins (date formatting, push notification language).

## Adding a new language

1. Create `src/i18n/<locale>/index.ts` with your translations
2. Pass it to `createSynkosBoot({ messages })`
3. Add the locale to `app.config.ts`:

```ts
defineAppConfig({
  i18n: {
    locales: [
      { code: 'en-US', name: 'English' },
      { code: 'es-ES', name: 'Español' },
      { code: 'fr-FR', name: 'Français' }, // ← new
    ],
  },
});
```

The language picker in settings will pick it up automatically.

## What happens when a key is missing

In dev: a console warning + the key is rendered as-is.
In prod: silent fallback to the default locale, key rendered if also missing.

We recommend running with `vue-i18n`'s missing-key handler in dev to catch issues early.

## Anti-patterns

**Don't hardcode user-visible strings.** Even one-off strings should go through `t()`. The cost is one line per string — the payoff is being able to ship to a new market without a code change.

**Don't put HTML in translations.** Use slot interpolation instead:

```vue
<i18n-t keypath="legal.agreeTo" tag="p">
  <template #terms>
    <a href="/terms">{{ t('legal.terms') }}</a>
  </template>
</i18n-t>
```

```ts
{ legal: { agreeTo: 'I agree to the {terms}', terms: 'terms' } }
```
